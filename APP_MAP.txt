================================================================================
  ROLES
================================================================================

  ADMIN      – verwaltet Lehrer/Bewerbungen via /admin; Auth: ADMIN_KEY-Env + admin_auth Cookie
  TEACHER    – erteilt Nachhilfe, verwaltet Verfügbarkeit, nimmt Buchungen an/ab
  STUDENT    – sucht Lehrer, bucht Stunden, chattet nach Buchung


================================================================================
  1. ADMIN SCENARIOS
================================================================================

  1a. Admin Login
  ---------------
  /admin/login
    -> enter ADMIN_KEY
    -> POST /api/admin/login
       [valid]   -> sets admin_auth HttpOnly cookie (24h) -> weiterleiten
       [invalid] -> Fehlermeldung

  1b. Admin Logout
  ----------------
    -> POST /api/admin/logout -> löscht admin_auth Cookie

  1c. Lehrerbewerbungen verwalten (NEUER FLOW – zweistufig)
  ----------------------------------------------------------
  /admin/applications
    -> Listet alle TeacherApplication-Einträge mit Statusbadge
    -> Status-Lifecycle:
         new  ->  contract_sent  ->  active  (| rejected)

    STUFE 1 – "Vertrag senden" (status === "new")
      -> POST /api/admin/applications/[id]/send-contract
         -> liest contracts/Lehrervertrag-LernApp.html
         -> füllt Name + E-Mail der Lehrkraft ein
         -> schickt HTML als Anhang an Bewerber-E-Mail
         -> setzt status = "contract_sent"
         Badge: gelb "Vertrag verschickt"

    STUFE 2 – "Vertrag bestätigt → Aktivieren" (status === "contract_sent")
      -> POST /api/admin/applications/[id]/confirm-contract
         -> erstellt Teacher-Datensatz aus Bewerbungsdaten (upsert by email)
         -> erstellt PasswordResetToken (24h)
         -> schickt Set-Password-E-Mail an Lehrkraft
         -> setzt status = "active"
         Badge: grün "Aktiv"

    "Erneut senden" (status === "contract_sent")
      -> POST /api/admin/applications/[id]/send-contract (idempotent)

    "Löschen"
      -> DELETE /api/admin/applications  { id }

    PDF ansehen (DSGVO-sicher, nur für Admin)
      -> GET /api/admin/applications/file?name=<filename>
         -> liest aus private-uploads/ (NICHT public/)
         -> prüft admin_auth Cookie
         -> liefert PDF mit Content-Type application/pdf

  1d. Lehrer manuell anlegen
  --------------------------
  /admin/teachers/new
    -> Formular: name, email, subject, schoolTrack, levelPref, schoolForms
    -> POST /api/admin/teachers
       -> erstellt Teacher (mustChangePassword=true)
       -> erstellt PasswordResetToken
       -> schickt Set-Password-E-Mail

  1e. Lehrer verwalten
  --------------------
  /admin/teachers
    -> GET /api/admin/teachers  -> Lehrerliste
    -> GET/PATCH/DELETE /api/admin/teachers/[id]  -> Einzellehrer
    -> POST /api/admin/teachers/[id]/resend-link  -> neuen Set-Password-Link schicken

  1f. Schüler verwalten
  ---------------------
  /admin/students
    -> GET /api/admin/students
    -> GET/DELETE /api/admin/students/[id]

  1g. Buchungen (Admin-Übersicht)
  --------------------------------
  /admin/bookings
    -> GET /api/admin/bookings
    -> GET/PATCH /api/admin/bookings/[id]

  1h. Fehler-Logs
  ---------------
  /admin/errors
    -> GET /api/admin/error-logs

  1i. Stats
  ---------
  GET /api/admin/stats  -> Kennzahlen (Buchungen, Umsatz, etc.)


================================================================================
  2. TEACHER ONBOARDING
================================================================================

  2a. Bewerbung einreichen (öffentlich, kein Login)
  --------------------------------------------------
  Landing page (/) oder /aplly  (TeacherApplySection-Komponente)
    -> Name, E-Mail, Fach, Motivationsschreiben, PDF-Upload (max 10 MB)
    -> POST /api/teachers/apply
       -> Rate-Limit: 3 pro IP pro Stunde
       -> speichert PDF in private-uploads/<timestamp>-<filename>.pdf
          (NICHT in public/ – kein öffentlicher Direktzugriff)
       -> speichert TeacherApplication in DB (status: "new")
       -> E-Mail an Admin: "Neue Lehrerbewerbung" (Link: /api/admin/applications/file?name=...)
       -> Bestätigungs-E-Mail an Bewerber

  2b. Passwort festlegen (nach Admin-Aktivierung)
  ------------------------------------------------
  /teacher/set-password?token=<token>
    -> Neues Passwort eingeben
    -> POST /api/teacher/set-password
       [gültig]    -> hasht Passwort, mustChangePassword=false, löscht Token
       [abgelaufen] -> Fehler; Admin muss neuen Link schicken

  2c. Teacher Login
  -----------------
  /auth/teacher/login
    -> E-Mail + Passwort
    -> next-auth "teacher-credentials" Provider
       [ok]    -> JWT Session -> /teacher/dashboard
       [fehler] -> Fehlermeldung

    -> GET/PUT /api/teacher/weekly-schedule

  Gesperrte Zeiträume (Urlaub, Krankheit):
    -> POST /api/teacher/unavailability  -> TeacherUnavailability (fromDate, toDate)
    -> DELETE /api/teacher/unavailability/:id

  3c. Profil
  ----------
  /teacher/profile

================================================================================
  4. STUDENT SCENARIOS
================================================================================

  4a. Registrierung / Login
  -------------------------
  /auth/login  (AuthModal-Komponente auf Landing Page)
    -> Registrierung: POST /api/register  -> erstellt User (Student)
    -> Login: next-auth "student-credentials" Provider -> JWT -> /student/dashboard
  /student/forgot-password  -> POST /api/student/forgot-password -> Reset-Link per E-Mail
  /student/set-password     -> POST /api/student/set-password

  4b. Profil
  ----------
  /student/profile
    -> Profilbild: POST /api/student/profile/picture
    -> Name, Adresse: PATCH /api/student/profile
    -> Schulinfos (read-only)

  4c. Lehrer suchen
  -----------------
  /student/dashboard
    -> Textsuche oder Fach-Chip
    -> GET /api/search?subject=...&studentEmail=...  -> passende Lehrer nach Schulprofil
    -> GET /api/teachers  -> alle Lehrer (inkl. avgRating, ratingCount, profilePicture)

  4d. Stunde buchen (vollständiger Stripe-Flow)
  ---------------------------------------------

  SCHRITT 1 – Lehrer ansehen, Termin wählen
  /book/:teacherId
    -> GET /api/teachers/:id  -> Lehrerprofil inkl. Bewertungen
    -> GET /api/student/teacher-blocked-dates?teacherId=...  -> gesperrte Zeiträume
    -> GET /api/student/teacher-booked-slots?teacherId=...&date=...  -> belegte Slots
    -> Dauer + Preis live berechnet: (Minuten/60) × 33 €
    -> "Jetzt buchen": POST /api/bookings
         -> Schüler-User upsert by email
         -> prüft Überschneidungen (pending/checkout_started/payment_method_saved/paid)
         -> erstellt Booking (status: "pending")

  SCHRITT 1b – FAGG-Pflichtinfo (§ 4 FAGG) ← NEU
    -> Vor Stripe-Weiterleitung: CheckoutDisclosureModal erscheint
       -> zeigt: Leistung, Termin, Gesamtpreis, Anbieter, Widerrufsbelehrung
       -> "Weiter zur Zahlung" oder "Abbrechen"
    -> POST /api/student/checkout { bookingId }
         -> Stripe Checkout Session (mode=setup, Karte speichern, noch kein Charge)
         -> Booking status -> "checkout_started"

  SCHRITT 2 – Stripe Checkout (hosted)
    -> Schüler gibt Kartendaten ein
    [abgeschlossen] -> Stripe Webhook: checkout.session.completed
                    -> /api/stripe/webhook/webhook
                    -> Booking: stripePaymentMethodId gesetzt, status = "payment_method_saved"
                    -> Weiterleitung zu /success
    [abgebrochen]  -> /cancel; Booking bleibt "checkout_started" (Cron löscht es)

  4e. Buchungsübersicht & Zahlung
  --------------------------------
  /student/payments
    -> GET /api/student/bookings  -> alle Buchungen
    -> "Jetzt zahlen": CheckoutDisclosureModal → Stripe
    -> "Stornieren" (vor Zahlung): POST /api/student/bookings/[id]/cancel
    -> Bewertung abgeben (nach "paid"): POST /api/ratings

  4f. Account löschen
  -------------------
    -> POST /api/student/account  (oder DELETE)


================================================================================
  5. TEACHER BOOKING ACTIONS
================================================================================

  /teacher/dashboard  (FullCalendar)
    -> GET /api/bookings/teacher?email=...
    -> Farben: BLAU=payment_method_saved, GRÜN=paid, ROT=declined/payment_failed,
               GRAU=pending/checkout_started/canceled_by_system

  5a. Annehmen
  ------------
  -> POST /api/bookings/confirm { bookingId }
     -> status muss "payment_method_saved" sein
     -> erstellt Stripe PaymentIntent (off_session, sofortige Abbuchung)
     -> IN TRANSAKTION: status="paid", stripePaymentIntentId gesetzt,
                        Availability-Slot gelöscht
     -> erstellt oder reaktiviert Chat (teacherId + studentEmail)
     -> E-Mail an Schüler: "Bestätigt, XX € abgebucht"
     [Zahlung fehlgeschlagen] -> status="payment_failed"

  5b. Ablehnen
  ------------
  -> POST /api/bookings/decline { bookingId }
     -> detacht stripePaymentMethodId bei Stripe
     -> status="declined", Stripe-IDs gelöscht
     -> E-Mail an Schüler

  5c. Lehrer storniert bestätigte Buchung
  ----------------------------------------
  -> POST /api/teacher/bookings/[id]/cancel
     -> Stripe Refund
     -> status="teacher_cancelled"
     -> E-Mail an Schüler

  5d. Automatische Ablaufstornierung (Cron)
  ------------------------------------------
  GET /api/cron/acceptDeadline  (externer Scheduler, z.B. Vercel Cron)
    -> Buchungen älter als 6 Tage mit status pending/checkout_started/payment_method_saved
    -> detacht Stripe-Karten, E-Mail an Schüler, status="canceled_by_system"


================================================================================
  6. CHAT
================================================================================

  Chat-Raum entsteht automatisch beim Annehmen (POST /api/bookings/confirm):
    -> upsert Chat (teacherId + studentEmail); ein Chat pro Paar

  Ungelesene-Badge:
    -> Chat.teacherLastRead / studentLastRead (DateTime)
    -> GET /api/teacher/unread-chats-count?email=...
    -> GET /api/student/unread-chats-count
    -> PATCH /api/chat/:chatId/mark-read { role: "teacher"|"student" }

  Teacher: /teacher/chat  ->  /teacher/chat/:chatId
    -> GET /api/teacher/chat  -> Chatliste
    -> GET /api/chat/:chatId/messages  (polling 1.2s)
    -> POST /api/chat/:chatId/messages  -> Nachricht senden (sender="teacher")

  Student: /student/chat
    -> GET /api/student/chat  -> Chatliste
    -> GET /api/chat/:chatId/messages
    -> POST /api/chat/:chatId/messages  -> Nachricht senden (sender="student")

  Sender-Typen: "teacher" | "student" | "system"  (system = gelbe Banner)


================================================================================
  7. TEACHER PAYOUT FLOW
================================================================================

  Revenue-Split: Schüler zahlt 100% → Plattform 40%, Lehrkraft 60%
  Auszahlung nur für abgehaltene Stunden (booking.end < now)

  7a. Stripe Connect Onboarding
  ------------------------------
  /teacher/payments
    -> POST /api/teacher/stripe-connect  -> Express-Account erstellen + Onboarding-URL
    -> GET  /api/teacher/stripe-connect  -> Status prüfen (charges_enabled + payouts_enabled)
       -> Teacher.stripeConnectOnboarded = true wenn ok

  7b. Auszahlung beantragen
  -------------------------
  -> POST /api/teacher/payout
     -> earnedCents   = Σ floor(priceCents × 0.6) für paid-Buchungen (end < now)
     -> paidOutCents  = Σ TeacherPayout.amountCents
     -> availableCents = earned – paidOut
     -> stripe.transfers.create → destination: stripeConnectAccountId
     -> erstellt TeacherPayout-Datensatz

  7c. Balance / History
  ---------------------
  GET /api/teacher/payout  -> earnedCents, paidOutCents, availableCents, payouts[]


================================================================================
  8. BOOKING STATUS LIFECYCLE
================================================================================

  pending
    └─> checkout_started  (Schüler öffnet Stripe Checkout)
          ├─> [abgebrochen] ──────────────────────> canceled_by_system (Cron)
          └─> payment_method_saved  (Webhook: Karte gespeichert)
                ├─> [Lehrer lehnt ab] ──────────> declined
                ├─> [6 Tage keine Aktion] ──────> canceled_by_system (Cron)
                └─> [Lehrer nimmt an]
                      ├─> [Zahlung schlägt fehl] -> payment_failed
                      └─> paid  (Karte abgebucht, Chat erstellt)
                            └─> paid + end < now  (60% für Auszahlung verfügbar)

    -> nur technisch notwendige Cookies (§ 165 Abs. 3 TKG 2021)
    -> Buttons: "Verstanden" + "Schließen" (kein Tracking-Consent erforderlich)

    -> zeigt Leistung, Preis, Anbieter, Widerrufsbelehrung

  10. DATEI-UPLOADS (DSGVO-sicher)
================================================================================

  Bewerbungs-PDFs:

    -> abrufbar nur via: GET /api/admin/applications/file?name=<filename>

    -> gespeichert in: public/uploads/profiles/
    -> öffentlich abrufbar (kein sensibles Inhalt)

  13. DATENMODELLE (Prisma / SQLite)
================================================================================

  User              Schüler (email unique, schoolTrack/Form/Level/Grade, Stripe-IDs)
  Teacher           Lehrer (subject String, schoolTrack, allowedForms JSON, Stripe Connect)
  TeachingOffer     Normalisiertes Angebot (teacherId, subjectId, schoolForm, level, grade-Range)
  Subject           Fach-Lookup-Tabelle
